# 项目初始化综述规则

## 规则简介

本规则指导如何分析项目并生成结构化的项目综述文件，帮助开发者快速了解项目全貌。生成的综述文件将写入 `.jarvis/rule.md` 文件，作为后续开发的上下文参考。

**核心原则**：综述必须**完备、准确、可操作**，覆盖项目全貌，不留盲区。如果综述文件已存在，必须先读取现有内容再进行增量更新，禁止直接覆盖。

## 你必须遵守的原则

### 1. 分析完备性原则

**要求说明：**

- **必须**：分析项目的目录结构，理解源码组织方式
- **必须**：识别项目的技术栈（编程语言、框架、依赖库及版本）
- **必须**：理解项目的架构设计和核心模块
- **必须**：确认项目的构建、运行和测试方式
- **必须**：识别关键配置文件和环境变量
- **必须**：统计代码规模（语言分布、文件数、代码行数）
- **必须**：识别项目的架构特点（设计模式、扩展机制、本地优先等）
- **必须**：梳理API与接口设计（公共API端点、协议、数据流）
- **必须**：识别项目的部署方式和运行环境要求
- **必须**：分析模块间的依赖关系和数据流向
- **禁止**：跳过任何核心模块的分析
- **禁止**：臆测项目信息，必须基于实际文件内容
- **禁止**：遗漏重要的配置文件或环境变量
- **禁止**：忽略测试和部署相关信息的收集

### 2. 综述结构化原则

**要求说明：**

- **必须**：使用 Markdown 格式组织综述内容
- **必须**：按章节组织（项目概述、技术栈、目录结构、核心模块、构建与运行、测试、关键配置、代码统计、架构特点、API与接口）
- **必须**：提供具体的命令示例（构建、运行、测试）
- **必须**：说明模块间的依赖关系
- **必须**：使用具体的版本号和数值，避免模糊描述
- **禁止**：使用模糊的描述（如"合理的架构"、"常见的框架"、"部分依赖"）

### 3. 增量更新原则

**要求说明：**

- **必须**：在写入综述文件前，先检查 `.jarvis/rule.md` 是否已存在
- **必须**：如果文件已存在，先使用 `read_code` 读取现有内容
- **必须**：基于现有内容进行增量更新，保留仍然有效的信息
- **必须**：标注更新时间和变更摘要（在文件顶部注明最后更新日期）
- **必须**：对比新旧信息，更新已过时的内容（如版本号、依赖变更）
- **禁止**：在已有综述文件时直接覆盖写入，丢弃已有分析成果
- **禁止**：忽略现有综述中的有效信息

## 你必须执行的操作

### 操作0：检查现有综述

**执行步骤：**

1. 检查 `{{ git_root_dir }}/.jarvis/rule.md` 文件是否存在
2. 如果存在，使用 `read_code` 读取现有综述的完整内容
3. 分析现有综述中哪些信息仍然有效、哪些需要更新
4. 记录需要新增的章节和需要更新的内容

### 操作1：浏览项目结构

**执行步骤：**

1. 使用 `execute_script` 工具执行 `find . -maxdepth 3 -type f -name '*.md' -o -name '*.json' -o -name '*.toml' -o -name '*.yaml' -o -name 'Dockerfile*' -o -name 'Makefile' | grep -v node_modules | grep -v __pycache__` 识别项目配置文件
2. 使用 `execute_script` 工具执行 `tree -L 2 -d` 或 `find . -maxdepth 2 -type d | grep -v '\./\.'` 查看目录结构
3. 识别主要目录的用途（src、tests、docs、config、scripts、ci、deployment等）
4. 使用 `execute_script` 工具执行代码统计命令（如 `cloc src/ --by-file` 或 `tokei`）
5. **关键文件清单**：必须检查以下文件（如果存在）：
   - 项目配置：`pyproject.toml`, `package.json`, `pom.xml`, `Cargo.toml`, `go.mod`, `CMakeLists.txt`
   - 构建工具：`Makefile`, `build.gradle`, `webpack.config.js`, `tsconfig.json`
   - 容器化：`Dockerfile`, `docker-compose.yml`, `.dockerignore`
   - CI/CD：`.github/workflows/*.yml`, `.gitlab-ci.yml`, `Jenkinsfile`, `.circleci/config.yml`
   - 环境配置：`.env.example`, `config/*.yaml`, `*.config.js`
   - 文档：`README.md`, `CONTRIBUTING.md`, `ARCHITECTURE.md`, `docs/` 目录

### 操作2：分析技术栈

**执行步骤：**

1. **读取主依赖配置**：使用 `read_code` 读取依赖配置文件的完整内容（如 `pyproject.toml`, `package.json`, `requirements.txt`, `pom.xml` 等）
2. **确定语言版本**：提取编程语言的版本约束（如 Python 3.12, Node.js 18+）
3. **识别核心框架**：列出主要框架及版本（Web 框架、测试框架、构建工具等），区分：
   - 核心运行时依赖（必须安装才能运行）
   - 开发依赖（仅开发时需要）
   - 可选依赖（扩展功能，如 `[browser]`, `[dev]` extras）
4. **检查子模块依赖**：如果项目是 monorepo 或有子模块，检查每个子模块的依赖配置
5. **验证依赖使用**：抽查 2-3 个核心依赖在代码中的实际 import/使用情况，确认依赖声明准确
6. **识别工具链**：记录使用的开发工具（linter、formatter、bundler、compiler 等）及其版本

### 操作3：理解核心模块

**执行步骤：**

1. **识别模块结构**：使用 `execute_script` 工具执行 `find src/ -type d -maxdepth 2` 列出源码目录结构
2. **读取入口文件**：读取主入口文件（如 `__init__.py`, `main.py`, `index.js`, `App.java` 等）的前 100 行，理解项目启动流程
3. **分析核心模块**：对于每个核心模块目录：
   - 使用 `read_code` 读取模块的主文件（如 `__init__.py`, `index.ts`）
   - 识别模块暴露的公共 API（导出的类、函数、常量）
   - 记录模块的职责和主要功能
4. **追踪模块依赖**：使用 `symbol_dependency` 工具（如果可用）或 `grep -r "^from\|^import" src/` 分析模块间的导入关系
5. **识别关键类和函数**：对于核心模块，列出最重要的 3-5 个类/函数及其用途
6. **理解数据流**：追踪核心数据结构在模块间的传递路径（如请求对象、配置对象、状态对象等）

### 操作4：分析构建与运行

**执行步骤：**

1. **查找构建脚本**：
   - 使用 `read_code` 读取 `Makefile`, `package.json`（scripts 部分）, `pyproject.toml`（tool.poetry.scripts 部分）
   - 识别构建命令（build, compile, bundle 等）
2. **确认运行方式**：
   - 识别主入口命令（如 `python -m module`, `npm start`, `java -jar`）
   - 检查是否有 CLI 工具（`setup.py` 的 `entry_points`, `package.json` 的 `bin`）
   - 记录启动参数和环境变量要求
3. **分析环境要求**：
   - 操作系统约束（Linux, Windows, macOS）
   - 运行时版本要求（Python 3.12, Node.js 18+）
   - 系统依赖（如 Clang, PostgreSQL, Redis）
   - 网络端口需求
4. **检查容器化支持**：
   - 读取 `Dockerfile` 和 `docker-compose.yml` 的前 50 行
   - 记录镜像构建命令和运行配置
   - 识别容器间的依赖关系（数据库、缓存、消息队列等）

### 操作5：梳理测试与部署

**执行步骤：**

1. **分析测试框架**：
   - 使用 `execute_script` 执行 `find tests/ -name '*.py' -o -name '*.test.js' -o -name '*_test.go' | head -20` 列出测试文件
   - 读取测试配置文件（`pytest.ini`, `jest.config.js`, `phpunit.xml` 等）
   - 识别测试类型（单元测试、集成测试、E2E 测试）
2. **确认测试命令**：
   - 从 `Makefile`, `package.json`, `pyproject.toml` 中提取测试命令
   - 记录测试运行方式（`pytest`, `npm test`, `go test`）
   - 识别测试覆盖率工具（coverage, nyc, jacoco）
3. **分析部署配置**：
   - 检查 CI/CD 配置文件（`.github/workflows/*.yml`, `.gitlab-ci.yml`）
   - 读取部署脚本（`deploy.sh`, `Jenkinsfile`）
   - 识别部署目标（云平台、容器编排、传统服务器）
4. **识别环境配置**：
   - 检查 `.env.example` 或 `config/` 目录
   - 列出关键环境变量及其用途
   - 识别配置管理方式（环境变量、配置文件、配置中心）

### 操作6：识别架构特点

**执行步骤：**

1. **识别架构模式**：
   - 单体应用 vs 微服务 vs 插件化架构
   - 分层架构（MVC, 三层架构, 六边形架构）
   - 事件驱动 vs 请求-响应模式
2. **分析通信机制**：
   - 同步通信：HTTP API, RPC, 方法调用
   - 异步通信：消息队列, WebSocket, 事件总线
   - 进程间通信：管道, 共享内存, Socket
3. **识别扩展机制**：
   - 插件系统（插件加载、注册、生命周期）
   - 配置驱动（配置文件、环境变量、远程配置）
   - 钩子机制（生命周期钩子、中间件、拦截器）
4. **分析关键设计模式**：
   - 创建型模式（工厂、单例、建造者）
   - 结构型模式（代理、适配器、装饰器）
   - 行为型模式（观察者、策略、命令）
5. **识别特殊架构特征**：
   - 分布式部署支持（Master/Child 节点、服务发现）
   - 认证授权机制（Token, OAuth, RBAC）
   - 缓存策略（内存缓存、分布式缓存）
   - 数据持久化（数据库、文件存储、对象存储）
6. **梳理组件交互流程**：
   - 选择 2-3 个典型业务场景（如用户请求处理、数据同步、任务调度）
   - 追踪每个场景的完整交互序列：入口 → 中间组件 → 存储层 → 返回路径
   - 识别关键交互路径和性能瓶颈点
   - 记录异常处理和容错机制
   - 使用文字描述或 ASCII 序列图说明组件间的调用关系

### 操作7：梳理 API 与接口

**执行步骤：**

1. **HTTP API 端点**：
   - 使用 `grep -r "@app\|@router\|@RestController" src/` 查找路由定义
   - 列出主要 API 端点、HTTP 方法、功能说明
   - 识别认证要求（公开 vs 需要认证）
2. **WebSocket 接口**：
   - 搜索 WebSocket 相关代码（`websocket`, `ws://`, `Socket.IO`）
   - 记录 WebSocket 端点、消息类型、数据流向
3. **内部接口与协议**：
   - 识别模块间通信接口（函数签名、数据结构）
   - 分析自定义协议（如节点间通信协议、任务调度协议）
   - 记录关键数据格式（JSON Schema, Protobuf, 自定义格式）
4. **事件系统**（如果有）：
   - 列出系统事件类型
   - 记录事件发布者和订阅者
   - 说明事件数据格式
5. **CLI 接口**（如果有）：
   - 列出命令行工具的子命令
   - 记录参数和选项
   - 说明典型使用场景

### 操作8：生成综述文件

**执行步骤：**

1. 基于前面收集的信息，按照综述模板组织内容
2. 如果是增量更新，合并新旧信息，标注更新日期和变更摘要
3. **确保完整性**：对照检查清单，确认所有必需信息都已收集
4. **验证准确性**：所有版本号、命令、路径都必须是实际存在的，不能臆测
5. 使用 `edit_file` 工具写入 `.jarvis/rule.md` 文件
6. 在文件顶部添加元数据：`> 最后更新：YYYY-MM-DD | 变更摘要：[简述本次更新内容]`
