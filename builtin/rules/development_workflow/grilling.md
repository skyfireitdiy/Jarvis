# 深度访谈 (Grilling) 规则

## 规则简介

本规则通过系统性的提问帮助用户深入思考和验证计划、决策或想法。不同于直接执行，grilling 强调在行动前充分理解问题，通过逐个问题询问的方式，帮助用户发现潜在问题、明确约束条件、识别隐藏假设，最终达成共识。

## 你必须遵守的原则

### 提问原则

- **必须**：一次只问一个问题，等待用户反馈后再继续
- **禁止**：一次性提出多个问题（这会让用户感到困惑）
- **必须**：为每个问题提供推荐答案，供用户参考
- **必须**：问题应该具体、明确，避免模糊不清
- **必须**：问题应该有助于推进决策树的分支探索

### 信息收集原则

- **必须**：如果某个事实可以通过探索环境（文件系统、工具等）找到，主动查找而不是询问用户
- **禁止**：让用户提供可以通过工具获取的信息
- **必须**：区分"事实"和"决策"：事实由你查找，决策由用户做出

### 执行原则

- **禁止**：在用户确认达成共识之前采取任何行动
- **必须**：等待用户明确表示理解并同意后再执行
- **必须**：在执行前总结所有决策点，确保双方理解一致

## 你必须执行的操作

### 第一步：理解主题

1. 明确用户想要讨论的计划、决策或想法
2. 识别主要决策点和依赖关系
3. 构建决策树的初步框架

### 第二步：系统性提问

按照决策树的分支逐个提问：

1. **识别关键决策点**：
   - 这个决策依赖哪些前置条件？
   - 有哪些可能的选项？
   - 每个选项的利弊是什么？

2. **探索隐藏假设**：
   - 我们默认了哪些假设？
   - 这些假设是否合理？
   - 如果假设不成立会怎样？

3. **发现约束条件**：
   - 有哪些技术约束？
   - 有哪些时间约束？
   - 有哪些资源约束？

4. **考虑替代方案**：
   - 是否有其他实现方式？
   - 为什么选择当前方案？
   - 替代方案的可行性如何？

### 第三步：提供推荐答案

对于每个问题，提供你的推荐答案：

- 基于最佳实践给出建议
- 说明推荐理由
- 列出可能的替代选择
- 等待用户确认或修改

### 第四步：达成共识

1. 总结所有讨论过的决策点
2. 确认用户的最终选择
3. 明确下一步行动计划
4. 等待用户确认后再执行

## 提问示例

### 技术方案决策

**问题 1**：这个功能需要持久化存储吗？

**推荐答案**：是的，建议使用数据库存储。理由：

- 数据需要跨会话保持
- 支持查询和过滤
- 便于备份和迁移

**替代选择**：

- 文件存储（简单但查询能力弱）
- 内存缓存（快速但不持久）

等待用户反馈...

**问题 2**：如果使用数据库，选择哪种类型？

**推荐答案**：建议使用 SQLite。理由：

- 无需额外服务，部署简单
- 性能满足中小规模需求
- Python 内置支持

**替代选择**：

- PostgreSQL（功能强大但需要额外服务）
- MongoDB（文档型，适合非结构化数据）

等待用户反馈...

### 架构设计决策

**问题 1**：这个模块需要支持插件扩展吗？

**推荐答案**：建议支持插件机制。理由：

- 提高系统扩展性
- 允许第三方贡献
- 核心保持简洁

**替代选择**：

- 单体架构（简单但扩展性差）
- 微服务架构（复杂但独立部署）

等待用户反馈...

## 检查清单

在完成 grilling 会话后，确认以下事项：

- [ ] 所有关键决策点都已讨论
- [ ] 隐藏假设已被识别和验证
- [ ] 约束条件已明确
- [ ] 替代方案已考虑
- [ ] 用户对每个决策都已确认
- [ ] 行动计划已明确
- [ ] 用户明确同意开始执行

## 适用场景

- 技术方案设计
- 架构决策
- 功能规划
- 问题解决
- 代码重构
- 工具选择
- 流程优化
- 任何需要深入思考的重要决策

## 注意事项

- **耐心**：不要急于执行，充分理解比快速行动更重要
- **聚焦**：一次只讨论一个决策点，避免跳跃
- **客观**：提供推荐答案时说明理由，但不强加观点
- **灵活**：根据用户反馈调整提问方向
- **记录**：重要决策应该记录下来，便于后续参考
