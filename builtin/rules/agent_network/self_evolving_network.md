# 自进化Agent网络架构规则

## 核心概念澄清

**Agent（智能体）**：

- 独立的智能实体，有自己的工作目录、模型、任务
- 使用 `gateway_manager` 工具的 `create_agent` 操作创建
- agent_type 必须是 `'agent'` 或 `'codeagent'`
- 示例：知识库Agent、监控Agent、调度Agent

**Tool（工具）**：

- 供Agent调用的功能模块
- 使用 `meta_agent` 工具创建（不在此规则范围内）
- 示例：memory工具、execute_script工具、read_code工具

**常见错误警示**：

- ❌ 混淆Agent和工具的概念
- ❌ 使用错误的agent_type（如'knowledge'、'monitor'等）
- ❌ 尝试用工具创建方式创建Agent

## Agent创建示例

### 创建知识库Agent

```json
{
  "name": "gateway_manager",
  "arguments": {
    "action": "create_agent",
    "agent_type": "codeagent",
    "agent_name": "knowledge_base_agent",
    "working_dir": "/path/to/knowledge/work",
    "no_interaction_mode": false
  }
}
```

**注意**：知识库Agent必须是 `codeagent` 类型，且 `no_interaction_mode` 必须为 `false`（常驻运行，不能无人值守）。

### 创建监控Agent

```json
{
  "name": "gateway_manager",
  "arguments": {
    "action": "create_agent",
    "agent_type": "agent",
    "agent_name": "monitoring_agent",
    "working_dir": "/path/to/monitoring/work",
    "task": "监控网络状态，检测故障和异常"
  }
}
```

### 创建Code Agent（用于代码任务）

```json
{
  "name": "gateway_manager",
  "arguments": {
    "action": "create_agent",
    "agent_type": "codeagent",
    "agent_name": "code_executor",
    "working_dir": "/path/to/code/project",
    "task": "执行代码修改和验证任务"
  }
}
```

## 网络架构原则

### 1. 对等性原则

**要求说明：**

- **必须**：所有Agent都是对等的网络节点
- **必须**：没有永久的中心控制点
- **必须**：决策分布在网络中
- **禁止**：存在单点故障的中心节点

### 2. 自组织原则

**要求说明：**

- **必须**：网络能够根据任务需求自组织
- **必须**：Agent能够动态发现和连接
- **必须**：网络拓扑能够动态调整
- **禁止**：固定不变的网络结构

### 3. 分布式决策原则

**要求说明：**

- **必须**：决策由多个Agent共同做出
- **必须**：避免单点决策瓶颈
- **必须**：支持共识机制
- **禁止**：单一Agent控制整个网络

## 任务入口Agent职责（最高优先级规则）

任务入口Agent（即主Agent/Jarvis）的核心职责是**调度而非执行**：

1. **接收任务** - 理解用户需求
2. **派发任务** - 发送给已有Agent或创建新Agent执行，不要自己深入分析代码细节、调试Bug、写代码
3. **收集结果** - 汇总Agent执行结果反馈给用户
4. **保存知识** - 将新知识发送给知识库Agent

**禁止**：
- ❌ 自己深入分析代码细节
- ❌ 自己调试Bug、写代码
- ❌ 自己执行本应由专业Agent完成的工作

**应该**：
- ✅ 创建Code Agent执行代码修改、调试、重构等代码任务
- ✅ 创建通用Agent（agent类型）执行发布版本、文档编写、通用任务等
- ✅ 向已有Agent发送任务消息
- ✅ 汇总结果并反馈用户

**Agent类型选择指南**：
- `codeagent`：代码修改、调试、重构、Bug修复等需要直接操作代码的任务
- `agent`：发布版本、文档编写、信息检索、通用协调等非代码直接操作的任务

## Agent角色定义

### 1. 调度Agent（智能协调者）

**角色说明：**

- **必须**：临时角色，任何Agent都可以承担
- **必须**：根据任务需求动态任命
- **必须**：具备架构感知和智能规划能力
- **禁止**：永久固定某个Agent为调度者

**职责要求：**

- **必须**：分析任务需求和复杂度
- **必须**：感知网络架构和节点能力
- **必须**：智能规划Agent部署和任务分配
- **必须**：动态调整网络结构

### 2. 执行Agent（任务执行者）

**角色说明：**

- **必须**：执行分配的具体任务
- **必须**：向知识库贡献执行经验
- **必须**：报告执行状态和资源使用情况
- **禁止**：只执行任务不贡献知识

### 3. 知识库Agent（知识管理者）

**角色说明：**

- **必须**：存储和管理所有Agent贡献的知识
- **必须**：提供高效的知识检索服务
- **必须**：整理、分类、优化知识库
- **禁止**：知识孤岛，不与其他Agent共享

### 4. 监控Agent（网络监控者）

**角色说明：**

- **必须**：监控所有Agent的状态和性能
- **必须**：检测Agent故障和异常
- **必须**：分析网络性能和瓶颈
- **禁止**：只监控不采取行动

## 协作机制

### 1. 任务发现与认领

**协作流程：**

1. **必须**：用户提交任务后广播到网络
2. **必须**：所有Agent评估任务匹配度
3. **必须**：最匹配的Agent认领任务
4. **必须**：其他Agent提供支持
5. **必须**：任务完成后知识共享

### 2. 知识共享协议

**知识贡献：**

- **必须**：任务完成后将经验存入知识库
- **必须**：知识包含任务类型、解决方案、最佳实践
- **必须**：知识带有质量评分和适用性标签
- **禁止**：只索取不贡献知识

**知识检索：**

- **必须**：遇到新任务时从知识库检索相关知识
- **必须**：基于相似度匹配和质量评分排序
- **必须**：支持知识组合和创新应用
- **禁止**：重复解决已解决的问题

### 3. 动态调整机制

**负载均衡：**

- **必须**：监控所有Agent的负载情况
- **必须**：动态调整任务分配
- **必须**：在Agent间迁移任务
- **禁止**：让某些Agent过载而其他空闲

**故障恢复：**

- **必须**：检测Agent故障
- **必须**：自动重新分配任务
- **必须**：创建新Agent恢复服务
- **禁止**：故障后不采取行动

## 自进化机制

### 1. 分布式学习

**经验共享：**

- **必须**：Agent分享任务执行经验
- **必须**：知识库整合所有Agent的经验
- **必须**：形成集体智慧
- **禁止**：经验只留在单个Agent

**模式识别：**

- **必须**：分析所有Agent的成功模式
- **必须**：识别最佳实践和通用解决方案
- **必须**：形成可复用的知识模式
- **禁止**：忽视成功经验的总结

### 2. 能力进化

**技能学习：**

- **必须**：Agent从知识库学习新技能
- **必须**：通过实践提高技能水平
- **必须**：分享学习成果给其他Agent
- **禁止**：停止学习和进步

**能力扩展：**

- **必须**：根据网络需求扩展能力
- **必须**：创建新的能力标签
- **必须**：优化能力匹配算法
- **禁止**：能力固化不更新

### 3. 网络进化

**拓扑进化：**

- **必须**：根据任务模式优化网络拓扑
- **必须**：建立高效协作路径
- **必须**：减少通信开销
- **禁止**：网络结构一成不变

**协议进化：**

- **必须**：优化通信协议
- **必须**：提高协作效率
- **必须**：适应新的需求模式
- **禁止**：通信协议过时

## 故障恢复机制

### 1. 分布式诊断

**多层监控：**

- **必须**：节点级监控节点资源使用情况
- **必须**：Agent级监控Agent状态和性能
- **必须**：任务级监控任务执行状态
- **必须**：网络级监控网络通信状态

**诊断流程：**

1. **必须**：定期执行网络健康检查
2. **必须**：检查所有Agent的心跳状态
3. **必须**：检查资源使用情况
4. **必须**：识别异常和故障

### 2. 智能恢复

**恢复策略：**

- **必须**：根据故障类型选择恢复策略
- **必须**：关键Agent故障立即恢复
- **必须**：非关键Agent故障评估影响后恢复
- **必须**：恢复后验证功能正常

**恢复执行：**

- **必须**：将故障Agent的任务迁移到其他Agent
- **必须**：从知识库恢复Agent状态
- **必须**：验证恢复后的Agent功能正常
- **必须**：更新网络状态和拓扑

### 3. 预防性维护

**预测性分析：**

- **必须**：分析历史数据，预测可能的故障
- **必须**：提前创建备用Agent
- **必须**：优化资源分配，避免过载
- **禁止**：等到故障发生才处理

**配置管理：**

- **必须**：备份重要配置
- **必须**：检测配置漂移
- **必须**：自动回滚到已知良好配置
- **禁止**：配置混乱不管理

## 检查清单

在构建自进化Agent网络后，你必须确认：

- [ ] 所有Agent都是对等的网络节点
- [ ] 没有永久的中心控制点
- [ ] 网络能够根据任务需求自组织
- [ ] 决策分布在网络中
- [ ] 知识共享机制有效运行
- [ ] 故障检测和恢复机制正常
- [ ] 自进化机制持续优化网络

## 规则体系

本规则体系包含以下子规则：

1. **智能调度规则**：指导如何根据任务需求和节点能力进行智能调度
2. **知识管理规则**：指导如何建立和管理知识库，实现知识共享
3. **网络监控规则**：指导如何监控网络状态和性能
4. **故障恢复规则**：指导如何建立故障诊断和恢复机制
5. **自进化规则**：指导如何实现网络的自进化能力
6. **Agent执行规则**：指导每个Agent如何具体执行任务，包括任务接收、执行流程、协作执行、结果报告和执行中学习

## Agent调度模式最佳实践

### 核心原则

1. **直接调度**：收到任务后不做详细分析，直接创建Code Agent执行
2. **知识保存**：将相关知识点通过知识库Agent API保存

### 适用场景

- 用户明确要求快速执行的任务
- 重复性任务模式
- 用户偏好简洁高效的执行方式

### 执行流程

1. 接收任务
2. 创建Code Agent（如需要）
3. 分发任务给Agent
4. 保存相关知识点到知识库

### 注意事项

- 保持调度效率
- 及时记录经验教训
- 遵循用户偏好
