# 告警管理规则

当用户明确表达进行快速验证、原型验证或概念验证时，你必须启用快速验证模式。

## 你必须遵循的工作流程

### 快速验证模式启用流程

1. **识别验证需求**：
   - 用户明确表达"快速验证"、"原型验证"或"概念验证"需求
   - 确认验证目标和范围

2. **启用快速验证模式**：
   - 在响应开头明确告知用户已启用快速验证模式
   - 提醒用户此模式的风险和限制

3. **执行快速验证**：
   - 优先保证验证速度
   - 忽略非关键警告和质量检查
   - 专注于核心功能可行性验证

4. **风险控制与建议**：
   - 建议用户在快速验证后进行完整质量检查
   - 明确告知此模式不适用于生产环境

## 你必须遵守的原则

### 快速验证优先原则

- **必须**：优先保证验证速度，而非代码完美性
- **必须**：忽略代码风格、命名规范、重复代码等非关键警告
- **必须**：仍需识别和报告严重安全漏洞和功能缺陷
- **禁止**：在生产环境或正式版本中使用此模式

### 风险控制原则

- **警告**：此模式下可能忽略重要质量问题
- **必须**：在响应开头明确告知用户已启用快速验证模式
- **必须**：提醒用户此模式的风险和局限性
- **必须**：建议用户在快速验证后进行完整质量检查

## 具体要求和规范

### 快速验证模式要求

**模式启用条件：**

- **必须**：用户明确表达"快速验证"、"原型验证"或"概念验证"需求
- **必须**：确认验证目标和范围
- **必须**：评估使用此模式的必要性和风险

**执行规范：**

- **必须**：专注于核心功能的可行性验证
- **必须**：跳过非关键的静态扫描警告
- **必须**：仍需保证基本的代码正确性和安全性
- **必须**：提供简洁直接的实现方案，不过度优化

**质量控制：**

- **必须**：识别和报告严重安全漏洞和功能缺陷
- **禁止**：忽略可能导致系统崩溃或数据丢失的严重问题
- **必须**：在验证速度与安全性之间保持平衡

## 实践指导

### 快速验证最佳实践

**何时使用快速验证模式：**

- **概念验证**：验证新技术或方案的可行性
- **原型开发**：快速构建功能原型
- **紧急修复**：需要快速验证修复方案
- **学习探索**：探索新的编程模式或算法

**使用注意事项：**

- **明确告知用户**：在响应开头清晰说明已启用快速验证模式
- **记录技术债务**：记录因快速验证而产生的技术债务
- **制定后续计划**：建议完整的质量检查和重构计划
- **保持安全意识**：绝不能忽略严重的安全问题

**平衡验证速度与质量：**

- **核心功能优先**：确保核心逻辑正确性
- **安全底线**：绝不妥协安全性和数据完整性
- **渐进改进**：在验证基础上逐步完善代码质量
- **及时重构**：验证完成后及时进行代码重构

- [ ] 用户明确表达了快速验证需求

- [ ] 已告知用户此模式的局限性和风险
- [ ] 已建议用户在验证后进行完整质量检查

- [ ] 专注于核心功能的可行性验证
- [ ] 忽略非关键警告，但保持安全底线
- [ ] 提供简洁直接的实现方案

在完成快速验证后，你必须确认：

- [ ] 已验证核心功能的可行性
- [ ] 已识别和报告严重安全问题
