# AI 友好型代码结构规范

## 规则简介

本规范定义了编写 AI 友好型代码必须遵守的结构化要求，旨在使代码更易于被 AI Agent 理解、分析和维护。遵循这些规范可以显著提升 AI 辅助编程的效率和准确性。

**重要前提**：在进行任何代码结构改造或优化之前，**必须确保有充分的测试用例覆盖**。测试覆盖是代码重构的安全保障，没有测试覆盖的代码改造是高风险行为。

## 你必须遵守的原则

### 1. 文件大小限制

**要求说明：**

- **必须**：单个源代码文件不超过 1000 行（不含空行和注释）
- **必须**：当文件超过 800 行时，应考虑拆分
- **禁止**：将无关的功能模块放在同一文件中
- **例外**：如果文件功能非常内聚（如核心算法、领域模型定义），拆分反而降低可读性，可保留原文件但应添加说明注释

**拆分策略：**

- 按功能模块拆分：不同业务逻辑分离到不同文件
- 按职责拆分：数据模型、业务逻辑、接口定义分别存放
- 按复用性拆分：通用工具函数提取到独立模块

**示例：**

```python
# ✅ 正确：文件职责单一，约 200 行
# user_service.py - 只处理用户相关业务逻辑
class UserService:
    def create_user(self, ...): ...
    def get_user(self, ...): ...
    def update_user(self, ...): ...

# ✅ 正确：工具函数独立成文件
# utils/validation.py - 只包含验证相关函数
def validate_email(...): ...
def validate_phone(...): ...
```

### 2. 函数长度限制

**要求说明：**

- **必须**：单个函数不超过 50 行（不含空行和注释）
- **必须**：当函数超过 40 行时，应考虑提取子函数
- **禁止**：在函数内嵌套超过 3 层的条件判断

**拆分策略：**

- 提取子函数：将独立逻辑块提取为私有方法
- 提前返回：使用 guard clause 减少嵌套
- 策略模式：复杂条件分支使用策略模式

**示例：**

```python
# ✅ 正确：函数职责单一，约 20 行
def process_order(order: Order) -> bool:
    """处理订单，返回是否成功"""
    if not validate_order(order):
        return False

    if not check_inventory(order.items):
        return False

    total = calculate_total(order)
    return charge_payment(order.user, total)

# ❌ 错误：函数过长，超过 80 行
def process_order(order):
    # 验证逻辑... (20 行)
    # 库存检查... (20 行)
    # 价格计算... (20 行)
    # 支付处理... (20 行)
    # 日志记录... (10 行)
    pass
```

### 3. 单一职责原则

**要求说明：**

- **必须**：每个类/函数只负责一个明确的职责
- **必须**：类名和函数名能准确反映其唯一职责
- **禁止**：在工具类中混杂业务逻辑

**示例：**

```python
# ✅ 正确：职责分离
class UserAuthenticator:
    """只负责用户认证"""
    def authenticate(self, ...): ...

class UserProfileManager:
    """只负责用户资料管理"""
    def update_profile(self, ...): ...

# ❌ 错误：职责混杂
class UserManager:
    # 既管认证，又管资料，还管权限...
    def login(self, ...): ...
    def update_profile(self, ...): ...
    def assign_role(self, ...): ...
```

### 4. 清晰命名规范

**要求说明：**

- **必须**：使用有意义的名称，避免缩写（除非是通用缩写）
- **必须**：变量名体现数据类型和用途
- **必须**：函数名体现操作和行为（动词 + 名词）
- **禁止**：使用 `a`, `b`, `temp`, `data` 等模糊名称

**示例：**

```python
# ✅ 正确：命名清晰
def calculate_monthly_revenue(orders: List[Order]) -> Decimal: ...
def find_active_users_by_department(dept_id: int) -> List[User]: ...
user_creation_timestamp: datetime
max_retry_count: int = 3

# ❌ 错误：命名模糊
def calc(data): ...
def find(id): ...
tmp = ...
info = ...
```

### 5. 适度注释

**要求说明：**

- **必须**：为公共 API 编写文档字符串（docstring）
- **必须**：为复杂算法编写实现思路注释
- **必须**：为非直观的代码编写"为什么这样做"的注释
- **禁止**：注释重复代码本身（"what"而非"why"）

**示例：**

```python
# ✅ 正确：注释解释原因
def retry_request(url: str, max_retries: int = 3) -> Response:
    """
    发送 HTTP 请求，失败时自动重试。

    使用指数退避策略，因为立即重试可能加重服务器负载。
    """
    for attempt in range(max_retries):
        # 指数退避：1s, 2s, 4s...
        # 避免请求风暴，给服务器恢复时间
        wait_time = 2 ** attempt
        time.sleep(wait_time)

        try:
            return http.get(url)
        except ConnectionError:
            if attempt == max_retries - 1:
                raise

# ❌ 错误：注释重复代码
i += 1  # i 加 1
return result  # 返回结果
```

### 6. 扁平嵌套结构

**要求说明：**

- **必须**：代码嵌套层级不超过 3 层
- **必须**：使用 guard clause（提前返回）减少嵌套
- **禁止**：深层嵌套的 if-else 或 try-except

**示例：**

```python
# ✅ 正确：扁平结构（最大嵌套 2 层）
def process_data(data: Optional[Dict]) -> Result:
    if data is None:
        return Error("Data is None")

    if "items" not in data:
        return Error("Missing items")

    for item in data["items"]:
        if not is_valid(item):
            continue
        process_item(item)

    return Success()

# ❌ 错误：深层嵌套（5 层）
def process_data(data):
    if data is not None:
        if "items" in data:
            for item in data["items"]:
                if is_valid(item):
                    try:
                        process_item(item)
                    except Exception:
                        pass
```

### 7. 正交设计

**要求说明：**

- **必须**：模块之间保持低耦合
- **必须**：每个模块有明确的输入输出边界
- **禁止**：循环依赖和隐式全局状态

**示例：**

```python
# ✅ 正确：正交设计
# 模块 A：只依赖接口，不依赖具体实现
class PaymentProcessor:
    def __init__(self, gateway: PaymentGateway):  # 依赖抽象
        self.gateway = gateway

    def process(self, amount: Decimal) -> bool:
        return self.gateway.charge(amount)

# ❌ 错误：耦合严重
class PaymentProcessor:
    def process(self, amount):
        # 直接依赖具体实现
        from alipay_sdk import Alipay
        alipay = Alipay(...)
        return alipay.charge(amount)
```

## 你必须执行的操作

### 操作 0：测试覆盖检查（强制）

**执行步骤：**

1. 在进行任何代码结构改造前，**必须先检查测试覆盖情况**
2. 运行现有测试，确认测试通过率 100%
3. 检查测试覆盖率，核心业务逻辑覆盖率应 ≥ 80%
4. 如果测试覆盖不足，**禁止进行代码结构改造**，应先补充测试用例
5. 记录测试覆盖情况，作为重构安全性的基准

**示例：**

```bash
# 运行测试并检查覆盖率
pytest tests/ --cov=src/ --cov-report=term-missing

# 确认测试通过
# 确认核心模块覆盖率 ≥ 80%
```

### 操作 1：代码审查时检查

**执行步骤：**

1. 统计文件行数（排除空行和注释），确认≤500 行
2. 统计函数行数（排除空行和注释），确认≤50 行
3. 检查嵌套层级，确认≤3 层
4. 审查命名是否清晰有意义
5. 确认注释解释了"为什么"而非"是什么"

### 操作 2：重构超长文件

**执行步骤：**

1. 识别文件中的不同职责模块
2. 将每个职责模块提取到独立文件
3. 在新文件中保持原有函数签名
4. 在原文件中导入并转发调用
5. 运行测试确保行为不变

### 操作 3：重构超长函数

**执行步骤：**

1. 识别函数中的独立逻辑块
2. 将每个逻辑块提取为私有方法
3. 为新方法选择清晰的名称
4. 在原函数中调用新方法
5. 运行测试确保行为不变

## 检查清单

在编写或审查代码后，你必须确认：

- [ ] **测试覆盖检查已通过**（测试通过率 100%，核心业务逻辑覆盖率 ≥ 80%）
- [ ] 所有源文件不超过 500 行（不含空行和注释）
- [ ] 所有函数不超过 50 行（不含空行和注释）
- [ ] 所有类/函数只承担单一职责
- [ ] 所有命名清晰且有意义
- [ ] 关键代码有解释"为什么"的注释
- [ ] 代码嵌套层级不超过 3 层
- [ ] 模块之间无循环依赖

## AI 友好度量维度

本规范围绕 **7 大关键维度** 展开，确保代码结构对 AI Agent 友好：

### 维度 1：上下文窗口适配性

**关注点**：代码体量与检索路径是否适配 AI 上下文限制

**核心指标**：

- 模块平均 Token 数 < 150K
- 单文件 Token 数 < 10K（数据层 ≤ 15K）
- 函数/方法平均 Token 数 < 500（复杂函数 ≤ 800）
- 依赖上下文开销：模块内 < 50K，跨模块 < 30K 且 ≤ 3 个外部模块
- 关键路径检索深度：文件数 ≤ 5，层级数 ≤ 4
- 接口定义 Token 占比：数据层 30–50%，业务层 10–20%，接口层 20–30%
- 有效语义注释密度 ≥ 60

### 维度 2：语义可理解性

**关注点**：命名与实现一致、注释消除歧义、隐式行为显式化

**核心指标**：

- 领域术语覆盖率 = 100%
- 命名-实现一致性评分 ≥ 85
- 命名歧义率 ≤ 5%
- 命名规范一致性 = 100%
- 契约式注释覆盖率 ≥ 95%
- 隐式行为显式化率 = 100%
- 业务规则硬编码率 ≤ 10%
- 代码圈复杂度均值 ≤ 10

### 维度 3：结构可导航性

**关注点**：模块职责、内聚度、依赖、目录、接口边界清晰

**核心指标**：

- 目录结构合规率 ≥ 95%
- 架构分层遵循度 = 100%（无违规跨层调用）
- 跨模块依赖清晰度 = 100%（依赖显式无循环）
- 模块内聚度系数 ≥ 2.0
- 模块职责单一性：对外接口 ≤ 10，子目录 ≤ 5
- 入口点收敛度 < 5
- 接口边界明确性 = 100%
- 扇入/扇出比 ≥ 1.0（公共模块豁免）

### 维度 4：变更局部性

**关注点**：功能内聚、修改影响可控、副作用显性化

**核心指标**：

- 业务功能调用链深度 ≤ 8 层
- 单点变更影响范围 ≤ 5 模块或 ≤ 10 文件
- 副作用显性化率 = 100%
- 副作用隔离度：纯函数 ≥ 60%，全局修改点 ≤ 10/模块
- 配置外置化率 ≥ 90%
- 逻辑-配置平衡度 ≤ 10%

### 维度 5：测试可验证性

**关注点**：可测试性设计到位、核心逻辑可测、边界异常覆盖

**核心指标**：

- 可测试性设计评分 ≥ 80
- 核心逻辑测试覆盖率 ≥ 80%
- 关键路径测试覆盖率 ≥ 90%
- 接口验证覆盖率 = 100%
- AI 生成代码验证覆盖率 = 100%

### 维度 6：模式一致性

**关注点**：架构模式、编码风格、错误处理与日志规范统一

**核心指标**：

- 编码规范合规率 ≥ 95%
- 分层/架构模式统一度 = 100%
- API 语义化程度 = 100%

### 维度 7：文档完备性

**关注点**：文档与代码同步、无漂移、示例可执行、变更可追溯

**核心指标**：

- 文档漂移检测通过率 = 100%
- 文档-代码一致性 ≥ 90
- API 文档完整率 = 100%
- 示例代码可执行率 = 100%
- 变更可溯性 ≥ 90%

## AI 友好度评分模型

### 评分公式

```
Score = 0.8 × Score_规则 + 0.2 × Score_AI统计
```

- `Score_规则`：7 大维度**规则型指标**加权分
- `Score_AI统计`：**AI Agent 统计型指标**样本平均分

### 维度权重

| 维度             | 权重 |
| ---------------- | ---- |
| 上下文窗口适配性 | 20%  |
| 语义可理解性     | 20%  |
| 结构可导航性     | 20%  |
| 变更局部性       | 15%  |
| 测试可验证性     | 10%  |
| 模式一致性       | 10%  |
| 文档完备性       | 5%   |

### 等级标准

| 等级     | 分数区间 | 含义                          |
| -------- | -------- | ----------------------------- |
| **S 级** | 90–100   | AI 原生友好，可全自动开发     |
| **A 级** | 75–89    | AI 友好优良，适合 AI 辅助开发 |
| **B 级** | 60–74    | 基础可用，需适度优化          |
| **C 级** | 40–59    | AI 开发困难，需重构           |
| **D 级** | 0–39     | 不适合 AI 开发                |

### CI 门禁通过条件

1. **无一票否决项**（AI 反模式）
2. 规则分 ≥ 80
3. AI 统计分 ≥ 75
4. 强门禁项 100% 通过
5. AI 统计指标**置信度 ≥ 80%**
   - 置信度 95%：正常门禁
   - 置信度 80%：仅告警，允许合并

## AI 反模式清单（一票否决项）

以下反模式**任意一项存在，不得评为 A 级及以上**：

1. **深层嵌套宏定义/元编程**：AI 难以静态分析，理解成本极高
2. **运行时动态代理无类型提示**：AI 无法推断实际类型和行为
3. **魔法字符串/数字硬编码无枚举映射**：语义不明确，AI 无法理解业务含义
4. **隐式事件发布无显式标注**：副作用隐藏，AI 无法追踪数据流
5. **全局可变状态滥用**：状态不可预测，AI 无法推理程序行为
6. **配置驱动复杂流程控制**：逻辑分散，AI 无法建立完整模型

## AI Agent 统计型指标

### 度量模式定义

1. **规则型指标**：通过 AST、静态扫描、Tokenizer、目录结构等**确定性规则**计算，100% 可复现，CI 强门禁

2. **AI Agent 统计型指标**：对**语义类、意图类指标**采用**随机抽样 + 概率统计 + 区间判定**，不逐行全量，以统计结果作为门禁依据

3. **实验型指标**：用于迭代校准、复盘，**不进 CI 门禁**

### 全局抽样与统计规则

所有 AI Agent 统计型指标统一遵循：

1. **抽样对象**：函数/方法/接口/类
2. **抽样比例**：**5%~10%**，且每模块**最少抽样 3 个**
3. **评分方式**：0~100 分连续评分
4. **统计口径**：
   - **样本平均分** = 总分 / 样本数
   - **样本合格率** = 评分≥阈值的样本数 / 总样本数

### 置信度规则

- 样本量 ≥ 5：**置信度 95%**
- 样本量 3~4：**置信度 80%**
- 样本量 < 3：**置信度 0%，不参与门禁**

### 区间判定规则

- ≥ 目标分：**通过**
- 目标分−5 ~ 目标分：**告警不阻断**
- < 目标分−5：**不通过**
