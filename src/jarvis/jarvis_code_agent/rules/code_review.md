# 代码审查规则

## 审查重点

1. **功能正确性**：
   - 代码是否实现了预期的功能？
   - 是否处理了边界情况和异常情况？
   - 是否有潜在的安全漏洞？

2. **代码质量**：
   - 代码是否清晰、易读？
   - 命名是否准确、有意义？
   - 是否有重复代码需要重构？

3. **性能考虑**：
   - 是否有性能瓶颈？
   - 算法和数据结构选择是否合理？
   - 是否有不必要的计算或资源消耗？

4. **可维护性**：
   - 代码结构是否合理？
   - 是否遵循项目的编码规范？
   - 是否有适当的注释和文档？

## 审查建议

- 保持建设性的反馈，关注代码而非个人
- 提出具体、可操作的建议
- 认可好的实践和设计
- 及时响应审查请求
- 对于复杂的变更，考虑分阶段审查

## 审查清单

- [ ] 代码符合项目的编码规范
- [ ] 没有明显的 bug 或逻辑错误
- [ ] 有适当的错误处理
- [ ] 有必要的测试覆盖
- [ ] 代码注释清晰、准确
- [ ] 没有安全漏洞
- [ ] 性能可接受
- [ ] 代码易于理解和维护
