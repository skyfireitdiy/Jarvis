# Git 工作流规则

## 提交规范

1. **提交信息**：
   - 使用清晰、描述性的提交信息
   - 第一行应该简洁地总结变更
   - 详细说明放在后续行

2. **提交粒度**：
   - 每个提交应该是一个逻辑单元
   - 相关的变更应该一起提交
   - 不相关的变更应该分开提交

3. **提交频率**：
   - 频繁提交，小步快跑
   - 完成一个功能或修复后立即提交
   - 不要积累大量未提交的变更

## 分支策略

1. **主分支**：
   - main/master 分支应该保持稳定
   - 只接受经过审查和测试的代码

2. **功能分支**：
   - 每个新功能创建独立分支
   - 分支名应该描述功能
   - 功能完成后合并到主分支

3. **代码审查**：
   - 所有代码变更都应该经过审查
   - 审查通过后才能合并
   - 及时响应审查意见

## Git 最佳实践

- 使用有意义的提交信息
- 保持提交历史清晰
- 定期同步主分支
- 解决冲突时保持代码质量
- 使用 .gitignore 忽略不必要的文件
- 不要提交敏感信息

## 检查清单

- [ ] 提交信息清晰、描述性
- [ ] 每个提交是一个逻辑单元
- [ ] 代码经过测试
- [ ] 代码经过审查
- [ ] 没有提交敏感信息
- [ ] .gitignore 配置正确
