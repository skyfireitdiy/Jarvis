# Jarvis vs ZeroClaw 对比报告

## 执行摘要

Jarvis 与 ZeroClaw 都属于 AI Agent 相关项目，但两者的核心定位并不相同。

- **Jarvis** 更偏向面向开发者的代码工程 Agent 平台，重点在于把 Agent 能力落入代码阅读、修改、验证、提交、规则复用和经验沉淀等研发流程。
- **ZeroClaw** 更偏向通用 Agent 的运行时底座，重点在于模型接入、渠道接入、工具扩展、记忆、安全、网关和运行时抽象。

因此，两者并不适合被简单理解为同类替代品。更准确的说法是：它们有交集，但核心优势位于不同层面。Jarvis 更强在研发工作流闭环，ZeroClaw 更强在运行时基础设施与多环境适配。

## 项目定位

### Jarvis

Jarvis 的整体结构更接近“开发者工作台”。它不是单纯提供对话式 Agent 能力，而是在通用 Agent 之上继续叠加了代码工程能力，例如代码分析、文件修改、构建验证、静态检查、Git 工作流、规则加载、方法论复用以及专项工具链。

从实际模块组织来看，Jarvis 的重点在于：

- 让 Agent 更稳定地参与代码任务
- 把研发规范、经验和流程纳入执行过程
- 为特定工程任务提供专门能力，如安全分析、C→Rust 迁移、Windows GUI 自动化等

### ZeroClaw

ZeroClaw 的整体结构更接近“Agent Runtime”。它更强调底层抽象能力，把模型提供商、消息渠道、工具、记忆、运行时、安全与网关拆分为可组合子系统，以便在不同环境、不同接入方式和不同部署目标下复用。

从架构意图来看，ZeroClaw 更适合承担以下角色：

- 作为通用 Agent 的运行底座
- 面向多渠道、多环境、多接入方式部署
- 在资源、性能、安全和模块边界方面保持较强控制力

## Jarvis 核心能力概览

Jarvis 的基础能力由通用 Agent 层提供，在此之上进一步形成面向代码任务的增强层。

### 1. 通用 Agent 与任务执行

Jarvis 提供基础的 Agent 执行能力，包括任务处理、上下文管理、工具调用、用户确认、会话保存与恢复等。这使它不仅能进行对话，还能围绕具体任务持续推进执行过程。

### 2. CodeAgent 与代码工程闭环

Jarvis 的核心差异化能力在于 CodeAgent。与一般的通用 Agent 相比，它更强调围绕代码仓完成任务闭环，主要包括：

- 代码上下文分析
- 文件读取与修改
- diff 展示
- 构建验证
- 静态检查
- Git 工作流辅助
- 影响范围分析
- worktree 支持

这使 Jarvis 更适合直接参与开发、修复、重构和审查类任务。

### 3. 规则系统

Jarvis 将规则系统作为执行过程的一部分，而不是仅作为文档存在。规则可以参与任务理解、执行约束和输出风格控制。这使 Jarvis 在“规范复用”和“团队经验固化”方面表现更突出。

### 4. 方法论系统

除了规则，Jarvis 还强调方法论复用。其目标不是只保存静态知识，而是把历史任务经验转化为可复用的执行方法，从而减少重复试错。

### 5. 工具与专项能力

Jarvis 通过工具系统连接文件读取、文件编辑、脚本执行、规则加载、记忆管理、Web 搜索等能力。同时，它还提供若干专项模块，例如：

- 面向 C→Rust 迁移的专项流程
- 面向代码安全分析的专项流程
- 面向 Windows GUI 自动化的专项能力
- 面向 Web 接入的 Gateway 能力

## ZeroClaw 核心能力概览

ZeroClaw 的重点不在于围绕某一类业务场景做深度封装，而在于提供更完整的 Agent 运行时底座。

### 1. Provider 抽象

ZeroClaw 对模型提供商进行了独立抽象。这样的设计更利于统一处理模型接入、切换、路由与扩展。

### 2. Channel 抽象

ZeroClaw 有明确的渠道子系统，用于对接不同通信入口。这也是它与 Jarvis 的一个重要区别：Jarvis 有网关和交互入口，但 ZeroClaw 更强调大规模渠道化接入能力。

### 3. Tool、Memory 与 Runtime

ZeroClaw 将工具、记忆和运行时作为独立子系统组织。这样的拆分方式使其更像底层平台，而不是单一场景应用：

- Tool 负责能力扩展
- Memory 负责状态与知识记忆
- Runtime 负责执行环境抽象

### 4. Security、Gateway 与 Observability

ZeroClaw 在安全、网关、可观测性方面也呈现出更底座化的结构：

- Security 更偏运行时控制与策略层
- Gateway 更偏统一接入层
- Observability 更偏系统级观测与追踪

这说明 ZeroClaw 的主要优势在于系统化底层能力，而不是代码工程工作流本身。

### 模块映射结论

- Jarvis 的强项不在通用运行时底座，而在面向代码工程的上层工作流封装。
- ZeroClaw 的强项不在代码工程闭环，而在运行时基础设施的通用性与模块化。
- 两者真正重叠的部分主要是模型接入、工具系统、记忆、网关以及部分安全能力。
- 两者真正拉开差距的部分主要是：
  - **Jarvis**：规则、方法论、代码工程闭环、专项工程能力
  - **ZeroClaw**：多渠道连接、运行时抽象、可观测性、安全底座、硬件与边缘扩展

## 适用场景对比

### 更适合选择 Jarvis 的场景

- 代码开发、重构、审查与修复
- 研发流程自动化与半自动化协作
- 需要规则沉淀、经验复用、规范执行的场景
- 需要专项工程能力，如 C→Rust 迁移、安全分析、Windows GUI 自动化

### 更适合选择 ZeroClaw 的场景

- 需要多渠道接入的通用 Agent 系统
- 对启动速度、资源占用、运行时可移植性要求较高的场景
- 边缘计算、长期运行、低资源部署场景
- 需要更完整运行时安全、网关与可观测性能力的场景

### 不宜简单混同的地方

- Jarvis 的重点不是构建一个极致轻量的通用 Runtime，也不是覆盖大量消息渠道。
- ZeroClaw 的重点不是围绕代码工程构建完整开发闭环，也不是把规则与方法论沉淀作为核心能力。

## 总结

Jarvis 和 ZeroClaw 代表了两种不同的 AI Agent 产品化方向。

- **Jarvis** 更像“开发者工作台”，重点是把 Agent 能力嵌入代码工程流程。
- **ZeroClaw** 更像“Agent 运行时底座”，重点是把 Agent 能力做成可组合、可移植、可扩展的基础设施。

如果从“功能是否一样多”来比较两者，容易得出失真的结论。更合理的比较方式，是回到它们各自的设计目标与适用场景：

- 当目标是研发流程、代码任务执行、规则复用和专项工程能力时，Jarvis 更有针对性。
- 当目标是构建通用 Agent 运行平台、对接多渠道、多环境部署并强调底座能力时，ZeroClaw 更有代表性。

因此，这份对比的最终结论不是“谁替代谁”，而是“两者站位不同、重心不同、适用场景不同”。
