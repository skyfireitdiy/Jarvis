# 在构建失败时把错误信息转成后续修复提示

## 使用场景

当代码已经改动成功，但构建验证没有通过时，系统不会只是简单地打印报错结束，而是会把构建失败信息整理成后续修复提示，推动下一轮修改朝正确方向收敛。

## 操作步骤

1. 启动代码代理并提交代码任务。
2. 如果本轮修改后的构建验证失败，先查看终端里显示的失败摘要。
3. 如果你没有选择关闭构建验证，系统会自动把这些错误信息转换为下一步修复参考。
4. 继续让系统执行后续修复，或根据这些错误信息手动调整你的任务说明。

## 你会看到的提示与反馈

- 构建失败后，终端会显示失败提示和错误摘要。
- 如果你没有切换到“只做基础静态检查”，系统会把这些错误作为后续修复的重要参考继续使用。
- 这样做的效果是：下一轮修复不再只是盲目继续，而是会优先针对这次构建失败暴露的问题收敛。

## 注意事项

- 这一步是自动衔接到后续修复流程的，不需要你额外复制粘贴错误日志给系统。
- 如果错误信息特别长，终端里通常只会显示摘要，而不是完整无限展开。
- 这类修复提示的质量依赖于当前构建错误是否明确；如果错误本身来自外部环境缺失，后续仍可能需要你补充环境条件说明。
