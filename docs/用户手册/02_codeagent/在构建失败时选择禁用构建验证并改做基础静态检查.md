# 在构建失败时选择禁用构建验证并改做基础静态检查

## 使用场景

当你的项目只能在特殊环境中构建，或者当前机器一直无法完成完整构建验证时，系统会在首次构建失败后给你一个选择：以后不再做完整构建验证，改为只做基础静态检查。

## 操作步骤

1. 正常提交一个会触发代码修改的任务。
2. 如果本轮构建验证失败，先查看终端里的失败摘要。
3. 当系统询问你是否要禁用构建验证、后续仅进行基础静态检查时，根据你的项目情况做选择。
4. 如果你确认禁用，系统会立即记住这个选择，并在后续直接改做基础静态检查。

## 你会看到的提示与反馈

- 首次构建失败时，终端会显示失败摘要，并明确提示你可以选择禁用构建验证，后续仅进行基础静态检查。
- 如果你接受这个方案，系统会马上切换到基础静态检查，并把结果继续显示在本轮输出里。
- 这个选择会被保存到当前项目，下次再运行同类任务时，系统可以直接沿用这个设置。
- 如果你不接受，系统会继续按构建失败分支处理，并把错误信息转成后续修复提示。

## 注意事项

- 这个选项适合构建环境受限、CI 环境不完整、依赖外部服务或本机无法还原完整构建条件的项目。
- 禁用后并不代表完全不检查代码，而是改为更基础的静态检查路线。
- 这是项目级设置；一旦保存，后续同项目运行通常会继续沿用，除非你后面重新启用完整构建验证。
