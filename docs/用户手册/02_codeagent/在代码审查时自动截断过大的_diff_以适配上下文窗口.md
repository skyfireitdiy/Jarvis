# 在代码审查时自动截断过大的 diff 以适配上下文窗口

## 使用场景

当一次改动涉及太多文件或 diff 内容过大，已经不适合完整塞进一次代码审查时，系统会自动对 diff 做截断和压缩展示，避免审查阶段因为内容过长而失效。

## 操作步骤

1. 正常运行代码代理并提交会产生较大改动的任务。
2. 当系统进入代码审查阶段时，如果本轮 diff 过大，系统会自动改用截断后的 diff 继续审查。
3. 你不需要手动裁剪补丁，也不需要重新输入命令。
4. 查看终端里的截断提示，再结合文件列表继续判断本轮审查结果。

## 你会看到的提示与反馈

- 这是代码审查阶段自动发生的，不需要额外输入命令。
- 如果 diff 被截断，终端会明确提示当前审查使用了裁剪后的 diff。
- 提示里通常会说明原始 diff 的规模、当前展示了多少内容，以及完整修改涉及哪些文件。
- 即使 diff 没有完整展开，系统仍会继续本轮审查，而不是因为内容过大直接放弃。

## 注意事项

- 截断的目的，是让系统在上下文窗口有限时依然能继续审查，不代表未展示的部分完全不重要。
- 如果你的改动特别大，看到截断提示后，最好再结合 Git 工具自行查看完整补丁。
- 这类情况常见于大重构、大批量文件调整或一次性修改很多文件的任务。
