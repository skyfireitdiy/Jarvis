# 启用或禁用代码修改后的构建验证

## 使用场景

如果你希望代码修改后自动进行构建验证，以便尽早发现明显错误，需要了解当前构建验证行为。根据当前源码行为，构建验证默认开启。

## 操作步骤

1. 先确认你的项目是否适合在当前环境下执行构建验证。
2. 当前源码中，构建验证默认返回开启状态。
3. 如果你在使用现有 CLI/运行流程时发现某些特殊环境选择了跳过构建验证，应以当次运行入口的实际行为为准。
4. 修改代码后，观察系统是否自动执行构建验证步骤。

## 你会看到的提示与反馈

- 在支持该流程的场景下，代码修改后会自动进入构建验证。
- 如果当前运行环境或入口关闭了这一步，可能只会进行基础静态检查或直接跳过。
- 构建验证失败时，通常会在后续反馈中体现，而不是静默忽略。

## 注意事项

- 就当前 `config.py` 行为来看，这不是一个已经暴露为独立配置键的稳定可开关能力说明文档，而是“当前默认行为说明”。
- 写文档时不要把它描述成“用户已经可以通过 config.yaml 任意开关”的能力，除非后续源码明确提供对应配置项。
- 如果你需要稳定控制构建验证是否执行，应优先看你正在使用的具体入口参数或运行模式，而不只看配置文件。
