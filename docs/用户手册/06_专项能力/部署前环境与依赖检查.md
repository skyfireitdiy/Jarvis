# 部署前环境与依赖检查

## 使用场景

当你准备部署开源项目，希望在正式执行前先检查环境是否满足要求，避免部署到一半才发现版本、依赖、磁盘或网络条件不合适时，可以让部署流程先执行环境与依赖检查。

## 操作步骤

1. 在部署任务中明确要求：先做环境检查，再进入正式部署。
2. 提供目标项目、版本和运行环境信息。
3. 让系统检查与你的部署目标相关的运行环境和依赖，例如：

   - Python
   - Node.js
   - Go
   - Rust
   - 虚拟环境或工具环境
   - 操作系统信息
   - 系统依赖
   - 磁盘空间
   - 网络连接

4. 如果你想直接说明任务，可以使用类似表达：`在部署 xx 项目之前，请先检查 Ubuntu 22.04 环境中的 Python、uv、系统依赖、磁盘空间和网络连接是否满足要求，并列出需要补齐的项。`
5. 根据系统给出的检查结果，先完成缺失项安装、升级或替代方案处理。
6. 环境满足后，再继续正式部署步骤。

## 你会看到的提示与反馈

- 系统会先给出当前环境是否满足部署要求的判断。
- 若发现版本不符、依赖缺失或环境不合适，会给出明确整改建议。
- 对 Python 项目，通常会特别关注 `uv` 和虚拟环境状态。
- 完成检查后，你会得到一份更适合继续部署的环境清单或整改建议。

## 注意事项

- 这通常是部署专家流程中的前置阶段，不是完全独立的通用命令。
- 环境检查不是可有可无的附加项，而是正式部署前的重要步骤。
- 如果你忽略这一步，后续部署失败概率会明显上升。
- 检查结论应结合你的目标环境理解，例如本机、容器、CI 或远程服务器。
- 环境信息越具体，检查项越准确；如果不说明系统版本、部署方式和运行用户，结果容易偏泛。
