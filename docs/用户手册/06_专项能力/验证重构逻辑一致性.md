# 验证重构逻辑一致性

## 使用场景

当你完成一轮重构后，想确认新代码是否仍然保持原有逻辑，而不是只看代码风格、文件结构或命名变化时，可以使用内置重构检查专家进行逻辑一致性检查。

## 操作步骤

1. 在任务中明确告诉系统：使用重构检查专家。
2. 提供可用于对比的提交号，以及你希望检查的重构范围。
3. 如有需要，再说明你特别关注的功能路径或风险点。
4. 如果你想直接发起任务，可以使用类似描述：`请使用重构检查专家，对比当前实现与 commit abc123 的差异，重点检查用户登录、权限判断和异常返回逻辑是否保持一致。`
5. 提交任务后，等待系统读取对应版本内容，并对当前逻辑进行比较分析。
6. 查看最终结论，确认是“逻辑一致”还是“存在不一致”。

## 你会看到的提示与反馈

- 系统会基于你提供的提交号读取可访问的历史版本内容。
- 在需要时，还会读取代码上下文辅助判断。
- 最终会给出逻辑一致性结论，并附带分析依据。
- 如果无法访问提交号或上下文不足，系统通常会先提示前提问题，而不是直接给出武断结论。

## 注意事项

- 这是专家型任务入口，不是固定 CLI 命令。
- 核心目标是校验“逻辑是否一致”，不是比较文本差异是否相同。
- 输入前提是仓库历史可访问，且你提供的提交号有效。
- 如果重构范围很大，建议分模块检查，避免一次结论过于笼统。
- 最好明确“基准版本 + 当前范围 + 重点功能路径”三项信息，否则分析结果容易停留在泛化结论。
