# 通过 commit 检查重构一致性

## 使用场景

当你已经知道要对比的 commit，希望明确基于该历史版本检查重构后的代码是否仍保持原有逻辑时，可以把 commit 作为主要输入来发起重构一致性检查任务。

## 操作步骤

1. 准备可访问的 commit 标识。
2. 在任务中明确说明：请基于该 commit 对比当前重构后的实现。
3. 指定你要检查的模块、文件或功能范围。
4. 如果你想直接给出清晰任务，可以使用类似描述：`请基于 commit abc123 对比当前 src/payment 模块的重构结果，重点检查订单创建、支付回调和异常处理逻辑是否保持一致。`
5. 提交任务后，等待系统读取目标 commit 的代码内容，并与当前实现进行逻辑对比。
6. 根据输出结果判断这次重构是否保持了原有功能逻辑。

## 你会看到的提示与反馈

- 系统会先读取指定 commit 对应的代码内容。
- 对比完成后，会输出是否逻辑一致的结论。
- 结果通常会更强调功能一致性和行为一致性，而不是格式或风格差异。
- 如果 commit 不可访问或仓库历史不完整，系统会提示前提条件不足。

## 注意事项

- 这类检查依赖仓库历史可访问，因此 commit 必须有效。
- 如果你的目标只是一般性的重构审查，也可以直接用“验证重构逻辑一致性”方式，而不必把重点放在某个 commit。
- 检查重点是功能逻辑，不是代码是否长得一样。
- 指定范围越清晰，分析结论通常越可信。
- 最好同时提供 commit 和当前检查范围；只给 commit 而不说关注模块时，输出往往会过于宽泛。
