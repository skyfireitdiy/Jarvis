# 在恢复会话前检查 Git commit 是否一致

## 使用场景

当你要恢复一个旧会话时，最大的风险之一就是当前代码已经不是当时保存会话时的那份代码了。系统会先做 commit 一致性检查，提醒你是否存在上下文错位风险。

## 操作步骤

1. 先回到原项目目录并启动 Jarvis。
2. 如果当前入口支持会话恢复，在对话输入框中输入 `'<RestoreSession>'`，再按 `Ctrl+J` 或 `Ctrl+]` 提交。
3. 选中要恢复的历史会话后，系统会先读取会话保存时记录的 commit 信息。
4. 然后系统会检查当前仓库的 HEAD commit。
5. 如果两者一致，就继续恢复；如果不一致，系统会先发出警告，再让你决定下一步。

## 你会看到的提示与反馈

- 如果一致，系统会提示：
  - `✅ Git commit 一致校验通过`
  - 并显示保存时和当前的 commit 缩写
- 如果不一致，系统会显示警告，并说明可能出现：
  - 代码上下文缺失
  - 引用的文件或函数不存在
  - 历史对话中的代码引用失效
- 非交互模式下，系统会直接提示：`🤖 非交互模式：自动继续恢复（状态可能不一致）`

## 注意事项

- 这一步不是多余检查，而是为了避免你在错误代码版本上继续旧任务。
- 如果你对当前仓库状态没有把握，不要忽略这个警告。
- commit 不一致时恢复出来的会话，可能会引用已经变化的代码和文件。
