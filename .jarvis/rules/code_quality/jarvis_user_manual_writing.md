# Jarvis 用户手册编写规则

## 规则简介

本规则用于约束 Jarvis 项目用户手册的编写与修订方式，确保文档真正面向最终用户，而不是面向开发者或维护者。

用户手册的核心目标不是“概括功能存在”，而是让用户**看完就知道怎么操作**。因此，所有文档都必须建立在真实源码行为、CLI 参数、交互提示、界面反馈和配置边界之上，禁止只根据功能标题、草案摘要或主观理解直接写正式文档。

## 你必须遵守的原则

### 1. 必须先阅读源码和真实交互行为

**要求说明：**

- **必须**：先阅读相关源码、CLI 入口、交互提示、配置项、前端界面或实际用户可见反馈，再开始编写正式用户手册。
- **必须**：优先确认用户真实会看到的提示文案、输入方式、提交方式、成功/失败反馈。
- **必须**：对自动触发、手动触发、条件触发、权限限制等行为进行区分。
- **禁止**：只根据功能标题、功能清单、草案摘要或函数命名推测用户行为。
- **禁止**：在未确认入口的情况下写出“先触发”“先启用”“调用入口”等模糊步骤。

### 2. 必须写成用户可直接操作的指南

**要求说明：**

- **必须**：每篇文档都尽量让用户可以直接照着做，而不是只理解概念。
- **必须**：操作步骤优先写清以下信息：
  - 用户在哪操作（终端、对话输入框、Web 界面、启动命令等）
  - 用户输入什么原样文本
  - 用户按什么键或如何提交
  - 系统会看到什么提示、列表、状态或结果
- **必须**：如果功能属于自动触发，要明确写出“这是自动发生的，不需要额外输入命令”。
- **禁止**：把“系统会……”当作主要步骤，却不说明用户前面需要做什么。
- **禁止**：写出用户无法直接执行的自我引用式步骤，例如“先查看”“先恢复”“先触发”，但不说明具体做法。

### 3. 必须保持面向用户，禁止泄露实现细节

**要求说明：**

- **必须**：使用最终用户能理解的语言描述功能、步骤和结果。
- **必须**：用“系统”“当前会话”“输入框”“提示”“列表”等用户视角词汇描述行为。
- **禁止**：在正式用户手册中暴露函数名、类名、内部调用链、源码文件路径、实现细节。
- **禁止**：出现“源码显示”“代码里可以看到”“根据某函数逻辑”等开发者视角表述。
- **禁止**：把开发者接口、内部工具调用方式直接当成普通用户入口，除非源码明确显示这是用户可见入口。

### 4. 必须写清边界条件和限制

**要求说明：**

- **必须**：对需要特定入口、特定配置、特定权限、特定模式才可用的功能写清边界。
- **必须**：明确区分交互模式与非交互模式、CLI 与 Web、自动行为与手动行为。
- **必须**：如果某命令不是所有入口都可用，要写成“如果当前入口支持……”。
- **必须**：如果某能力依赖预先开启的配置，要写成“先进入已经开启该能力的会话或启动方式”。
- **禁止**：默认假设所有启动方式都支持同一能力。
- **禁止**：把仅在任务结束阶段出现的提示，写成普通输入阶段可以随时触发的命令。

### 5. 必须统一文档结构并保证可验收

**要求说明：**

- **必须**：正式用户文档至少包含以下部分：
  - 使用场景
  - 操作步骤
  - 效果说明或你会看到的提示与反馈
  - 注意事项
- **必须**：文档风格保持统一，标题应描述用户能完成的动作或场景。
- **必须**：若是可交互能力，操作步骤中应尽量包含真实提示、快捷键、确认方式或输出结果。
- **必须**：若是自动能力，效果说明中应包含用户可观察到的提示、状态或后续影响。
- **禁止**：只有概括说明，没有用户步骤。
- **禁止**：把内部设计说明写成用户手册正文。

## 你必须执行的操作

### 操作1：编写用户手册前先做取证

**执行步骤：**

1. 确认目标功能对应的源码入口、CLI 参数、交互提示、配置行为或前端表现。
2. 记录用户实际可见的输入方式、快捷键、提示语、状态变化和结果。
3. 判断该功能是：手动触发、自动触发、条件触发，还是仅在某些入口可用。
4. 只有在关键信息确认后，才开始编写正式文档。

**注意事项：**

- 不明确的行为宁可继续取证，也不要凭经验补写。
- 如果真实入口尚未确认，优先在文档中写清前提边界，而不是编造统一入口。

### 操作2：把功能写成可执行步骤

**执行步骤：**

1. 先写使用场景，说明用户为什么会用这个功能。
2. 再写操作步骤，优先按“在哪输入 → 输入什么 → 如何提交 → 后续看到什么”的顺序组织。
3. 接着写效果说明或提示反馈，列出用户可能看到的真实信息。
4. 最后补充注意事项，说明限制、风险、入口差异和适用边界。

**注意事项：**

- 如果文档里出现“先查看”“先触发”“先启用”等词，必须继续补足具体动作。
- 如果某步骤是跳过、取消、回车、选择编号，也要写清楚。

### 操作3：完成后按用户可操作性回查

**执行步骤：**

1. 通读文档，检查用户是否能在不看源码的前提下直接照做。
2. 检查是否已经写清：输入位置、输入内容、提交方式、反馈提示。
3. 检查是否误写了开发者视角内容或内部实现细节。
4. 检查是否遗漏了模式差异、权限限制或自动/手动边界。

**注意事项：**

- 若读完文档后，用户仍然不知道第一步具体怎么做，则文档不合格。
- 若文档虽然正确但只停留在概括描述，也不合格。

## 检查清单

在完成用户手册编写或修订后，你必须确认：

- [ ] 文档内容已基于真实源码、CLI、交互提示、前端行为或配置行为取证
- [ ] 文档包含“使用场景 / 操作步骤 / 效果说明或提示反馈 / 注意事项”
- [ ] 操作步骤已尽量写清输入位置、原样文本、提交方式和用户可见反馈
- [ ] 自动触发与手动触发已明确区分
- [ ] 权限限制、入口限制、模式差异已写明
- [ ] 文档中未暴露函数名、内部调用链、源码依据字样或其他实现细节
- [ ] 文档从最终用户视角可理解、可执行、可验证
